# 感冒退烧 OTC AI 导购系统 Spec

## 背景

传统 OTC 药店导购依赖店员经验，存在三个问题：
- **安全风险**：店员缺乏系统医学训练，可能忽视高危症状，延误就医时机
- **推荐偏差**：推荐受商业利益影响，而非最适合用户症状的药
- **信息不完整**：用户不清楚药品的副作用、禁忌、药物相互作用，自行购药存在隐患

本系统定位为 **OTC 辅助推荐系统**——不是 AI 医生，不诊断疾病，而是为用户提供安全、专业、透明的购药引导。

**核心使用场景**：用户走进药店，知道自己不舒服但不知道买什么药。用户用自然语言描述症状，系统通过动态追问收集信息、排查危险信号、推荐合适的 OTC 药品。

**关键交互特性**：真实对话不是线性的。用户可能：
- 描述症状中途突然问「布洛芬有什么副作用？」
- 拿着药盒直接问「这个药我能吃吗？」
- 推荐完药后追问「有没有便宜一点的？」
- 任何时候说「算了我去医院」

系统必须具备**话题动态切换能力**——Graph 根据用户意图路由到对应节点，而非固定流程。

MVP 阶段聚焦 **感冒与退烧** OTC 品类。

## 目标

1. **症状理解与动态问诊**：用户用自然语言描述不适，AI 动态追问关键信息，不依赖固定问卷
2. **话题自由切换**：支持用户在任何节点跳转话题（问药、切换症状、放弃问诊），Graph 根据意图动态路由
3. **安全兜底**：推荐药物前规则引擎强制检查高危信号，发现危险引导就医
4. **对症推荐**：基于完整症状信息匹配最合适的 OTC 药品，解释推荐理由
5. **透明告知**：推荐药品时展示药物作用、副作用、禁忌
6. **库存闭环**：推荐后查库存，告知是否有货、价格、位置

## 功能需求

### F1: 匿名会话管理

用户进入药店终端后，系统自动创建匿名会话。该会话承载后续所有对话的上下文和历史。

- 创建会话时生成唯一 session_id
- 会话保留完整对话历史（用户消息 + AI 回复 + 中间状态）
- 会话有过期时间（默认 30 分钟无活动自动关闭）
- 会话关闭后可重新开始新会话

### F2: 用户意图识别与路由

每次用户输入后，系统识别用户意图，Graph 据此决定路由到哪个处理节点。MVP 支持以下意图：

| 意图 | 触发示例 | 路由目标 |
|------|---------|---------|
| 描述症状 | "我头疼流鼻涕" | Consult 节点 |
| 询问药品 | "布洛芬怎么吃"、"这个药有副作用吗" | Explain 节点 |
| 更换药物 | "有没有便宜点的"、"还有其他药吗" | Recommend 节点 |
| 切换症状 | "不对，我其实主要是咳嗽" | Consult 节点（重置部分 slot） |
| 放弃/就医 | "算了去医院" | 结束会话，输出就医引导 |
| 闲聊/其他 | "谢谢"、"你是机器人吗" | 通用回复 |

意图识别由 LLM 完成，不依赖关键词匹配。对于不确定的意图，系统追问澄清。

### F3: 动态问诊（Consult 节点）

用户描述症状后，系统以 ReAct 模式动态追问，直至信息充分或用户主动终止。

**追问维度**（ReAct Agent 自行判断需要哪项，不按固定顺序）：

| 维度 | 说明 |
|------|------|
| 症状细节 | 部位、性质、严重程度、开始时间 |
| 时间线 | 持续多久、何时开始、是否反复 |
| 已采取措施 | 是否已服药、剂量、效果 |
| 特殊人群 | 是否孕妇、哺乳期、儿童、老人 |
| 慢性病史 | 肝肾疾病、胃溃疡、哮喘等 |
| 过敏史 | 药物过敏史 |
| 其他症状 | 是否伴随发热、咳嗽、咽痛等 |

**退出条件**：
- 信息充分：ReAct Agent 判断可用于药物匹配
- 用户主动终止：「不用了」「去医院」
- 达到最大追问轮数：默认 6 轮（可在配置中调整）

### F4: 安全规则检查（SafetyCheck 节点）

推荐药物前，规则引擎强制执行安全检查。这是**确定性规则**，不依赖 LLM 判断。

**MVP 规则列表**（感冒退烧场景）：

| 规则 ID | 条件 | 动作 |
|---------|------|------|
| R1 | 体温 ≥ 39°C 且持续 ≥ 3 天 | 阻断推荐，引导立即就医 |
| R2 | 年龄 < 3 个月且发热 | 阻断推荐，引导立即就医 |
| R3 | 孕妇且体温 ≥ 38.5°C | 阻断推荐，引导就医 |
| R4 | 出现呼吸困难、胸痛、意识模糊 | 阻断推荐，引导急诊 |
| R5 | 全身皮疹/严重过敏反应 | 阻断推荐，引导就医 |
| R6 | 用户自述对推荐药物成分过敏 | 从候选列表中排除该药物 |
| R7 | 儿童（<12 岁）与含阿司匹林药物 | 排除含阿司匹林药物 |

**规则执行时机**：
- SafetyCheck 前：进入 Recommend 节点前，规则引擎判定是否允许继续
- Recommend 节点内：生成候选药物列表后，R6、R7 过滤不合适的药物

**结果**：PASS（继续推荐）/ BLOCK（阻断并给出就医理由）/ FILTER（排除特定药物）

### F5: 药品推荐（Recommend 节点）

安全规则检查通过后，系统根据已收集的症状信息匹配并推荐 OTC 药品。

**推荐逻辑**：
- 输入：Consult 节点输出的症状摘要（结构化 slot）
- 匹配维度：症状适配度、成分对症覆盖、剂型偏好（如有）
- 输出：1-3 个推荐药品（排序），每个附带推荐理由

**推荐约束**：
- 仅推荐 OTC 药品（处方药不在推荐范围内）
- 不推荐信息不充分时的"猜测性推荐"
- 候选药品必须经过 R6/R7 规则过滤
- 每个推荐附带一句话推荐理由

### F6: 药品解释（Explain 节点）

用户查询具体药品时，系统通过 RAG 检索药品说明书并结构化呈现。

**输出结构**：

| 字段 | 来源 | 说明 |
|------|------|------|
| 药品名称 | DB | 通用名 + 常见商品名 |
| 作用类别 | DB | 解热镇痛类、抗过敏类等 |
| 适应症 | DB + RAG | 对症说明 |
| 用法用量 | DB | 成人/儿童/老人 |
| 不良反应 | RAG | 常见/偶见/罕见 |
| 禁忌 | RAG | 绝对禁忌 + 慎用情况 |
| 药物相互作用 | RAG | 与其他药物合用时的风险 |
| 注意事项 | RAG | 饭后服用、避免饮酒等 |

**RAG 调用策略**：只在 Explain 节点调用，不在其他节点消耗 RAG 资源。

### F7: 库存查询（Inventory 节点）

推荐完成后，查询 PostgreSQL 中对应药品的库存信息。

**输出**：

| 字段 | 说明 |
|------|------|
| 药品名称 | 通用名 |
| 在售商品 | 对应药店的商品名、规格、厂家 |
| 库存状态 | 有货 / 缺货 / 库存紧张 |
| 价格 | 零售价 |
| 货架位置 | 如 "A区-3排-5层" |
| 替代建议 | 如缺货，推荐同成分替代药品 |

## 非功能需求

### N1: 响应延迟
- 意图识别 + 路由决策：< 1 秒
- 追问响应：< 3 秒（含 LLM 推理）
- 安全规则检查：< 100ms
- 库存查询：< 200ms
- RAG 检索 + 生成：< 3 秒
- 首次请求到首字节（TTFB）：< 5 秒

### N2: 安全与合规
- 所有对话历史不包含用户个人身份信息（手机号、身份证等），除非用户主动输入
- 药品推荐附带免责声明：「本系统仅为辅助参考，请仔细阅读说明书并按说明使用或在药师指导下购买和使用」
- 危险症状阻断后，系统强制输出就医引导，不允许绕开规则
- API 调用日志记录每次 SafetyCheck 的判定结果，用于审计

### N3: 可扩展性
- 新增 OTC 品类时，只需：新增药品数据 + 新增品类专属安全规则（可选）+ 新增对应药品说明书文档入 RAG
- ReAct 追问逻辑应通用化，不硬编码特定品类的追问模板
- 规则引擎支持插件式新增规则，不修改核心引擎代码

### N4: 可观测性与链路追踪
- 每个 Graph 节点的输入/输出记录到日志
- LLM 调用记录延迟和 token 消耗
- 规则引擎每次判定记录：触发了哪条规则、输入数据、判定结果
- **接入 LangSmith**：全链路 Trace 追踪，可视化每一次对话在 Graph 各节点的流转路径、延迟、LLM token 消耗

### N5: 部署与运维
- 所有基础设施（Milvus、PostgreSQL）通过 Docker Compose 一键启动
- FastAPI 应用提供 `/health` 健康检查端点，返回各依赖服务的连接状态
- 配置通过环境变量管理（LLM API Key、数据库连接串、LangSmith API Key 等）

## 不做的事

| 编号 | 内容 | 留给后续的原因 |
|------|------|--------------|
| O1 | 用户注册/登录/个人档案 | MVP 验证核心链路，用户系统是独立模块 |
| O2 | 处方药（Rx）推荐 | 法律风险，OTC 是安全边界 |
| O3 | 多轮对话中的图片识别（如用户拍药盒） | 多模态需要额外的模型和审核能力 |
| O4 | 药物购买/支付/订单 | 本系统是导购而非交易系统 |
| O5 | 感冒退烧以外的 OTC 品类 | 单一品类验证架构，后续扩展 |
| O6 | 知识图谱集成 | 当前 RAG + DB 已满足 MVP 需求，后续迭代引入 |
| O7 | 语音输入/输出 | 纯文本对话，语音是交互层扩展 |
| O8 | 多语言 | 仅支持中文 |
| O9 | 前端 UI | 专注后端 API，前端不在本次范围 |

## 验收标准

| 编号 | 验收标准 | 验证方式 |
|------|---------|---------|
| AC1 | 首次请求自动创建会话，返回 session_id | 调用 API，检查响应 |
| AC2 | 用户说"我头疼流鼻涕"，系统识别为「描述症状」意图 | 单元测试覆盖所有意图类型 |
| AC3 | 用户先说"头疼"又说"布洛芬有用吗"，系统自动从 Consult 切换到 Explain | 集成测试：模拟跳转场景 |
| AC4 | 用户说"算了去医院"，系统结束会话并输出就医引导 | 集成测试 |
| AC5 | ReAct Agent 追问至少覆盖 4 个不同维度后才结束 Consult | 集成测试 + LangSmith Trace 验证 |
| AC6 | 体温 ≥ 39°C 持续 3 天 → 阻断推荐，输出就医引导 | 规则引擎单元测试 |
| AC7 | 孕妇 + 发热 → 阻断推荐 | 规则引擎单元测试 |
| AC8 | 用户自述对布洛芬过敏 + 布洛芬在候选列表 → 自动排除 | 规则引擎单元测试 |
| AC9 | 症状完整输入后，输出 1-3 个推荐药品 + 推荐理由 | 集成测试 |
| AC10 | 不充分症状时（如只说"不舒服"），不输出推荐 | 集成测试 |
| AC11 | 查询"布洛芬副作用"，RAG 检索后输出结构化说明 | RAG 模块测试 |
| AC12 | 推荐药品后查库存，返回有货/缺货 + 价格 + 位置 | 集成测试 |
| AC13 | 免责声明出现在每次推荐药品的回复中 | 检查回复文本 |
| AC14 | 每个 Graph 节点转换在 LangSmith 可见完整 Trace | 运行一次完整对话后检查 LangSmith |
