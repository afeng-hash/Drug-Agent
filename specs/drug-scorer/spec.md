# 药品评分排序模块 (Drug Scorer) Spec

## 背景

当前系统在 `recommend_node` 中用 LLM（大语言模型）对候选药品排序。LLM 的问题在于：结果不可复现、排序逻辑黑盒、无法审计溯源、Prompt 微小的改动可能导致完全不同的排序结果。药店场景下，药师和监管机构需要明确的"为什么推荐 A 而不是 B"。

药品推荐本质是一个**多维度特征加权评分**问题——症状匹配度、安全性、年龄适配、成分覆盖等维度天然可量化。应该用确定性算法而非概率模型来做决策。

## 目标

- 用**加权线性模型**替代 LLM 做药品排序决策，确保结果确定、可复现、可审计
- 用**证据推理**作为特征生成器，每条证据独立影响对应特征值，实现打分过程可追溯
- 建立**三层权重管理体系**：策略层（语义约束）→ 配置层（DB 存储，可热更新 + A/B）→ 执行层（纯函数计算）
- 输出**评分明细**而不仅是排序：每个药品的最终得分可分解到维度级别的贡献

## 功能需求

### F1: 特征生成器 (Feature Generator)

系统将 `consult_slots`（症状槽位）和候选药品信息输入，通过一组独立的证据规则（Evidence），为每个药品生成结构化的特征向量。

每条证据规则只做一件事：观察输入中是否存在某种匹配/不匹配关系，设置对应特征值。

特征向量至少包含以下维度：
- `symptom_match`：症状与药品适应症的匹配程度（0-1）
- `safety`：安全性评分，综合禁忌症、过敏史、慢性病史（0-1，0=存在禁忌）
- `age_suitability`：药品对用户年龄段的适用程度（0-1）
- `otc_safety_level`：OTC 类别安全等级（乙类 > 甲类）
- `ingredient_coverage`：有效成分对症状的覆盖度（0-1）
- `evidence_quality`：支撑该推荐的信息完整度（0-1）

新增特征维度只需新增一条 Evidence 规则，不改其他代码。

### F2: 加权评分引擎

接收特征向量和权重配置，计算每个药品的线性加权得分：

```
score = Σ(wᵢ × featureᵢ)，其中 Σwᵢ = 1.0
```

- 权重必须归一化（总和为 1.0），确保不同版本配置的得分可比较
- 计算过程无状态、无IO、无随机性——纯函数
- 输出每个药品的**得分明细**：总分 + 各维度得分（wᵢ × fᵢ）

### F3: 安全降级机制

当某个药品的 `safety` 特征值低于阈值时（如存在禁忌症），该药品直接排除或降序到推荐列表末尾，不参与正常排序。

### F4: 三层权重管理

**策略层**：定义语义约束（如"安全优先策略下 safety 权重不低于 0.35"），负责校验配置是否合规。

**配置层**：权重值存储在 PostgreSQL 中，支持：
- 热更新：修改后实时生效（无需重启服务）
- 版本化：每次变更记录版本号 + 变更原因 + 操作人 + 时间
- A/B 测试：按 session 分桶路由到不同权重版本，LangSmith 记录版本标签

**执行层**：纯函数 `compute(features, weights) → ScoreResult`，不做 IO。

### F5: 评分可解释性

每次推荐输出评分明细，格式：

```
布洛芬 总分: 0.82
  symptom_match:  0.30 × 0.90 = 0.270  (症状"头痛"命中适应症)
  safety:         0.25 × 0.95 = 0.238  (无禁忌症，无过敏)
  age_suitability: 0.20 × 0.85 = 0.170  (成人，适用)
  otc_safety:     0.10 × 0.80 = 0.080  (甲类 OTC)
  ingredient_cov: 0.10 × 1.00 = 0.100  (布洛芬直接覆盖头痛+发热)
  evidence_quality: 0.05 × 0.80 = 0.040 (症状信息较完整)
```

每一项可追溯到具体证据来源。

### F6: 替代 LLM 排序

`recommend_node` 中的 LLM 排序调用替换为 Scorer 模块调用。LLM 仍保留用于生成推荐理由文案（`match_reason`），但不参与排序决策。

## 非功能需求

### N1: 确定性
相同输入必须产生相同输出。评分过程不依赖随机数、不依赖 LLM、不依赖外部服务时序。可复现、可验证。

### N2: 性能
单个药品评分耗时 < 1ms。12 个候选药 × 1ms ≤ 12ms。不得成为推荐链路的性能瓶颈。

### N3: 可扩展性
新增一个评分维度只需：(1) 注册一条 Evidence 规则，(2) 在配置表中添加对应权重。不改评分引擎代码、不改特征生成器框架。

### N4: 可观测性
每次评分输出结构化明细（维度级得分 + 证据来源），集成到 LangSmith trace 中。A/B 测试时按 `config_version` 聚合统计。

### N5: 配置热更新
权重配置从 DB 加载，TTL 缓存 60 秒自动刷新。运营修改权重后最多 60 秒内全局生效，无需重启、无需重新部署。

### N6: 测试覆盖
评分核心逻辑（特征生成 + 加权计算）的单元测试覆盖率 ≥ 95%。每条 Evidence 规则有独立单测。

## 不做的事

- **不做降级链**：DB 是权重配置的唯一来源。不提供文件兜底、不提供硬编码默认值
- **不做 LLM 辅助评分**：评分过程完全确定性，不引入 LLM 做任何评分相关的决策
- **不做实时权重回滚**：版本化管理支持手动回滚（切换 active_version），不做自动回滚
- **不做多策略自动切换**：MVP 阶段使用单一策略，场景自动识别留到后续迭代
- **不做权重自动学习**：不从用户反馈/点击率自动调整权重。权重由领域专家人工设定

## 验收标准

- **AC1**: 相同 `consult_slots` + 相同药品列表 → 两次评分结果完全一致（确定性）
- **AC2**: 布洛芬过敏用户 + 候选含布洛芬 → 布洛芬 safety < 阈值，被排除或降序
- **AC3**: 12 个药品评分耗时 < 15ms（性能）
- **AC4**: 新增一条 Evidence 规则并注册 → 不改框架代码即可生效（可扩展性）
- **AC5**: 修改 DB 权重值后 60 秒内 → 新请求使用新权重（热更新）
- **AC6**: 每次评分输出包含维度级明细 → 可追溯到每条证据的具体贡献（可解释性）
- **AC7**: A/B 分桶 → 同一 session 的两次请求使用同一版本权重
- **AC8**: `recommend_node` 不再调用 LLM 做排序，但 LLM 仍用于生成推荐理由文案
- **AC9**: 所有 Evidence 规则和评分引擎有独立单测，覆盖率 ≥ 95%
